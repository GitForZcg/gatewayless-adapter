# 适配器使用规则



## 一、作用

用于对接三方接口的数据适配转化服务，与业务服务解耦，框架包括统一签名处理、数据转化、数据校验、数据分发，工具类等。

## 二、数据流向模式

适配器分为两种数据流向模式：

1. 我方服务-----> 适配器-----> 三方  【<font color=green>**正向适配**</font>】
2. 三方-----> 适配器-----> 我方服务  【<font color=blue>**反向适配**</font>】

## 三、相关规范

### 1.正向适配相关规范

#### 1.1、正向适配统一接口

```java
@RestController
@RequestMapping("/adapter")
public class AdapterUnifiedController {

    private final BaseFlowManagerFactory flowManagerFactory;

    @Autowired
    public AdapterUnifiedController(BaseFlowManagerFactory paymentFlowManager) {
        this.flowManagerFactory = paymentFlowManager;
    }

    @PostMapping("/execute")
    public Object executePFlow(@RequestBody @Valid BaseUnifiedRequest request) throws Exception {
        return flowManagerFactory.executeFlow(
                request.getType(), request.getNode(), request.getParams()
        );

    }

}
```

##### 1.2.2、枚举统一参数

ServiceType: 表示我方的调用适配器服务的类型


BaseNode： 服务调用节点顶级规则，具体节点实现该接口, <font color=red>**新增服务节点枚举必须实现该接口**</font>

```java
public interface BaseNode {

    String name();
    String desc();
```

##### 1.2.5、业务数据体字段校验

校验字段分组，以**ServiceType**区分，orderGroup代表<font color=red>**ORDER服务的所有字段都要以orderGroup分组**</font>进行校验，其他服务同理。其中前缀为external 为反向适配的字段校验组。

```java
public interface ValidationGroups {
    interface orderGroup {
    }

    interface payGroup {
    }

    interface externalDemoProductGroup {
    }
}
```

业务数据体字段校验注解使用：只能加在一定 **不为空、不为null、不为""** 的字段上

以ORDER服务字段举例 ：格式为

<font color=blue>@**NotBlank**</font>(message = "", groups = <font color = red>ValidationGroups.**orderGroup**.class</font>)

<font color=blue> @**NotNull**</font>(message = "", groups = <font color = red>ValidationGroups.**orderGroup**.class</font>)

<font color=blue>@**NotEmpty**</font>(message = "", groups = <font color = red>ValidationGroups.**orderGroup**.class</font>)

这里要明确 @NotBlank  @NotNul @NotEmpty 的区别。

嵌套对象需要额外加@Valid 注解 否则无法校验到 嵌套对象中的字段内容

```java
@Data
@Accessors(chain = true)
public class TestOrderReqDto {

    @NotEmpty(message = "订单id不能为空", groups = ValidationGroups.orderGroup.class)
    private String orderId;

    @NotEmpty(message = "金额不能为空", groups = ValidationGroups.orderGroup.class)
    private String amount;
  
  	@NotEmpty(message = "子订单信息不能为空", groups = ValidationGroups.orderGroup.class)
    @Valid
    private SubOrder subOrderList;
}
```

##### 1.2.6、数据主体类命名规范

内部服务--->适配器服务的数据主体类以正常的Dto结尾命名

建议 由适配器访问外部服务的数据主体类以**ReqDto，RespDto**结尾，能明确区分出是 请求的数据 还是响应的数据主体类。

最好除了后缀之外的名称保持一致： 步骤如下

①ORDER请求--->适配器  数据转化 OrderQueryDto  ---> OrderQueryReqDto

③适配器请求--->外部服务 OrderQueryReqDto 

②外部服务响应--->适配器  数据转化OrderQueryRespDto----> OrderQueryResultDto

④适配器响应--->ORDER 服务  OrderQueryResultDto

#### 1.3、正向适配统一转化参数加签

##### 1.3.1、加签验签方法

由于外部服务加签规则不一致，需要加签的算法也<font color= red>不一致</font>

这里以内部ORDER服务 访问外部订单服务为例：

通过注入private final AbstractOrderMD5Service service; 来获取签名并发送请求方法，和验签解析数据方法。

AbstractOrderMD5Service在这里只适用于内部ORDER服务，请不要套用于其他服务请求，其他服务会提供同类型的加签验签的方法。

```java
public interface AbstractOrderMD5Service {
    <T extends DemoOrderMd5Param> Map<String, Object> execute(T reqDto, BaseNode node);
    <T extends DemoOrderMd5Param> T executeCheck(Map<String, Object> response, Class<T> clazz);
}
```

##### 1.3.2、请求加签数据类

举例：内部ORDER需要按照外部服务的MD5的方式生成签名，并按照外部服务的要求传输报文。

ORDER 请求外部服务的订单查询接口。
可以定义请求类为DemoOrderReqDto **必须** implements  DemoOrderMd5Param 

这里面的DemoOrderMd5Param与AbstractOrderMD5Service需要配套使用。

```java
@Setter
@Getter
@Accessors(chain = true)
public class DemoOrderReqDto implements DemoOrderMd5Param {

    private String aceOrderId;

    private String aceOrderAmount;

    /**
     * 需要加签的属性参数
     */
    public static final Set<String> NEED_SIGN_PARAMS = ImmutableSet.of("aceOrderId", "aceOrderAmount");

    @Override
    public Set<String> needSign() {
        return createSign(NEED_SIGN_PARAMS);
    }

    @Override
    public Set<String> needSignParam() {
        return null;
    }

    @Override
    public String orderId() {
        return createOrderId(this.aceOrderId);
    }
}
```

#### 1.4、正向适配统一响应参数

统一返回参数格式为Object，可通过demo-adapter-dto 中util 下的TypeHelperUtil 处理,其中包括了List,Map,以及自定义包装类的type转换

复杂对象转化**不建议以下写法，因为要写太多次数**

```java
 private final Type type = new TypeToken<List<Object>>(){}.getType();

//等同于 
 TypeHelperUtil.listOf(Object.class);
```

业务数据在处理完成时是知道具体的数据对象类型，其他服务调用适配器返回响应的内容时，通过TypeHelperUtil 转化返回的数据的数据对象类型即可。

以ORDER服务查询订单举例。请求对象为 TestOrderParams，响应的对象为 List<OrderQueryParams>

以下是伪代码：

```java
@Resource 
AdapterUnifiedApi  adapterUnifiedApi；

@PostMapping
public List<OrderQueryRespDto> orderQuery(){
		 TestOrderReqDto testReqDto = new TestOrderReqDto()
                .setOrderId("1234566645ss")
                .setAmount("10.00");
      OrderParams<TestOrderReqDto> orderParams = new OrderParams<>();
      orderParams.setBizData(testReqDto);
      BaseUnifiedRequest.Builder params = BaseUnifiedRequest.builder()
              .type(ServiceType.ORDER)
              .node(OrderNode.ORDER_QUERY)
              .params(orderParams);
  		//请求适配器处理
      Object obj = adapterUnifiedApi.executePFlow(params);
  		//转化成业务数据
     List<OrderQueryRespDto> result = GsonFactory.getInstance().fromJson(GsonFactory.getInstance().toJson(obj), TypeHelperUtil.listOf(OrderQueryRespDto.class));
     
     return result;
}
```



### 2.反向适配相关规范

#### 2.1、反向适配无接口配置

所有由非我方的服务的 外部服务进入的适配器的请求，全部为<font color=red>**无接口**</font>处理

接口定义全部为配置解析匹配，配置存储在whitelist_config 表中。

开发人员需要关注的配置字段如下:

api_path：外部服务访问适配器的接口地址，对外提供的接口地址 例如 /adapter/queryProduct

api_type：外部服务进入适配器后，后续适配器访问的我方服务类型，例如PRODUCT类型

api_sub_type：我方服务类型的子类型 ，通过子类型 （例如PRODUCT_QUERY）可以定位到我方内部服务接口

service_name:  后续适配器访问的我方服务的名称

class_name ：例如com.personal.demo.pojo.dto.TestDemoRequest，外部服务进入适配器验签后得到的**数据实体的类路径** 

<font color = green>建议将外部服务的实体类，定义在适配器的service服务里。适配器dto只放我方内部服务的数据实体类</font>

#### 2.2、反向适配接口业务数据类定义

##### 2.2.1、数据主体类命名规范

适配器服务--->内部服务的数据主体类以正常的Dto结尾命名，最好除了后缀之外的名称保持一致

反向适配对于数据主体类的规则比较多，建议由外部服务进入到适配器服务的文件名定义以**Request、Response**结尾。能明确区分出是 请求的数据 还是响应的数据主体类。

通过数据类后缀命名定义规则，可以区分出数据实体类 处于哪一种流向模式

最好除了后缀之外的名称保持一致： 步骤如下

①外部服务请求--->适配器 数据转化 OrderPushRequest  ---> OrderPushRequesDto

②适配器请求--->ORDER服务 OrderPushRequesDto  

③ORDER 服务响应--->适配器  OrderPushResultDto  

④适配器响应--->外部服务  OrderPushResultDto----> OrderPushResponse

### 3.异常使用规范

正向适配异常：

内部服务--->适配器---> 外部服务 出现的业务异常 ，以继承wrapper-base 包的规范为准。

反向适配异常：

外部服务--->适配器 AdapterException

适配器--->内部服务  以继承wrapper-base 包的规范为准。

#### 5.3、RedisKey统一生成管理工具

自定义枚举 实现implements RedisKeyGenerator ，里面定义了统一的**前缀后缀**拼接规范

以下举例

```java
@Getter
public enum RedisAdapterKey implements RedisKeyGenerator {
    CACHE_REQUEST_PATH("adapter:whitelist:request", "白名单"),
    CACHE_ACCESS_KEY("adapter:access:key", "访问密钥"),
    CACHE_FLOW_MAPPING("adapter:flow:mapping", "流程映射"),
    CACHE_DOMAIN("whitelist:domain:resolve", "域名缓存"),
    CACHE_DOMAIN_REVERSE("whitelist:domain:reverse", "域名解析缓存"),
    ;
    /**
     * -- GETTER --
     * 获取 Redis Key 的模板
     */
    private final String prefix;
    private final String desc;

    RedisAdapterKey(String template, String desc) {
        this.prefix = template;
        this.desc = desc;
    }

    @Override
    public String getPrefix() {
        return prefix;
    }
}
```

具体使用

```java
// 不通过拼接后缀生成key  adapter:access:key
String key = RedisAdapterKey.CACHE_ACCESS_KEY.generateKey();

// 通过拼接单个后缀生成key adapter:access:key_123456
String key = RedisAdapterKey.CACHE_ACCESS_KEY.generateKey("123456");

// 通过拼接多个后缀生成key adapter:access:key_123456_abcdef_ABCDEF
String key = RedisAdapterKey.CACHE_ACCESS_KEY.generateKey("123456","abcdef","ABCDEF");

//简化写法
String key =CACHE_ACCESS_KEY.generateKey("123456","abcdef","ABCDEF");

```
